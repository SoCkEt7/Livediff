# 14-day launch calendar

## Day 0 — Prep

- Confirm CI is green.
- Confirm `cargo install livediff` works.
- Confirm demo link loads.
- Add or verify GitHub topics.
- Keep Reddit/dev.to drafts ready but unpublished.

## Day 1 — GitHub polish

- Update About description if needed.
- Add high-intent topics.
- Add “When to use Livediff” to README.
- Create one release note or announcement post.

## Day 2 — dev.to / Hashnode

- Publish `devto-article.md` on one platform first.
- Set canonical URL.
- Reply to comments within 24h.

## Day 3 — Reddit technical post

- Pick one community only.
- Use the community-specific draft.
- Stay available for replies.

## Day 4 — Follow-up

- Turn the best comment thread into a README FAQ improvement.
- Open issues for valid feedback.

## Day 5 — Rust/TUI angle

- Share a technical note about `ratatui`, filesystem events, or ignore handling.

## Day 6 — Rest / monitor

- Check GitHub traffic/referrers.
- Do not repost the same pitch.

## Day 7 — Second community

- Post a different angle to a second community only if rules allow.

## Day 8-10 — Improve conversion

- Add examples from real workflows.
- Add screenshots or GIF variants.
- Improve install docs from feedback.

## Day 11 — Comparison content

Draft: “Live diff monitor vs rerunning git diff: when each workflow works better.”

## Day 12 — Contributor invitation

Create 2-3 small issues labeled `good first issue` with clear context.

## Day 13 — Release recap

Publish a short changelog-style post with what changed after feedback.

## Day 14 — Review metrics

Track:

- GitHub stars and traffic;
- demo visits if analytics exist;
- comments/questions;
- installs/downloads if available;
- repeated feature requests.

Use the data to pick the next content topic.
